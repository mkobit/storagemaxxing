## 1. Executive Summary

StorageMaxxing is a browser-based spatial planning tool that lets users model physical storage spaces — drawers, shelf bays, cabinets, entryway cubbies, garage areas — and see what filling them with different organizational systems would look like. The output is a set of visual layout options across multiple systems, a bill of materials, and a cost estimate, updated live as constraints change.

**Development is anchored to drawers first.** The drawer packing workflow — define dimensions, pick a system (Schaller bins, Gridfinity, or custom), apply constraints, see a live layout — is the core loop that proves out every primitive: space geometry, system catalog, constraint engine, options comparison, BOM. Everything else builds on top of it.

The core insight: most people have spaces they know are underutilized but have no good way to envision what's possible with the systems that actually exist. A standard tool-chest drawer could be filled with Schaller bins, tiled with a Gridfinity baseplate, or lined with custom acrylic — and those options look completely different. StorageMaxxing shows all of them, side by side, before the user commits to buying anything.

The problem is almost entirely unsolved by existing software. CAD tools can model space but have no catalog awareness or packing logic. Vendor planning tools (where they exist at all) are single-product and trivially simple. Logistics packers solve the math but are inaccessible to non-technical users.

> **Vision:** Describe your space. Pick a system or ask to see options. Get a visual layout with a real BOM and real prices — for every viable approach — so you choose based on evidence, not guesswork.

## 2. Problem Statement

### 2.1 What Users Actually Do Today

Without a purpose-built tool, planning storage is fragmented and error-prone:

- Measure spaces with a tape measure, sketch on paper or in a notes app
- Browse vendor websites (Schaller, Akro-Mils, Lista, Husky, Amazon) in separate tabs
- Manually verify fit: divide shelf width by bin width, check depth, repeat for each bin type
- Attempt to visualize the arrangement in their head — often getting it wrong
- Build a spreadsheet BOM by hand, frequently getting quantities wrong
- Order, discover incompatibilities, return, re-order

For mixed-use spaces — an entryway organizer, a garage wall, a workshop — users also face a more fundamental problem: **they don't know what options even exist.** A shelf with 14″ of clearance might fit a single row of tall bins, two tiers of small bins, a hanging rail system, or a custom Gridfinity grid. Without a tool that surfaces these alternatives, users either default to the first thing they find on Amazon or leave the space unoptimized.

### 2.2 The Gap

| Tool                    | Space Modeling             | Catalog Awareness               | Auto-Pack         | Options Exploration     | BOM      |
| ----------------------- | -------------------------- | ------------------------------- | ----------------- | ----------------------- | -------- |
| CAD (Fusion, SketchUp)  | Full                       | None                            | None              | None                    | None     |
| Schaller Drawer Builder | Fixed 24×24 only           | Schaller only                   | None              | None                    | None     |
| Vendor websites         | None                       | Single vendor                   | None              | None                    | None     |
| Logistics packers       | Abstract volumes           | None                            | Partial           | None                    | Partial  |
| **StorageMaxxing**      | **Sketch + dims + import** | **Multi-vendor + open systems** | **Live reactive** | **Multiple strategies** | **Live** |

### 2.3 The Real Problem Is Optionality

The v0.1 PRD framed StorageMaxxing as "optimize a known space with known bins." This is too narrow. The more powerful framing — and the one that distinguishes this from everything else — is **"show me what's possible in a space I have."**

A user with an entryway cubby doesn't arrive knowing they want Akro-Mils 30-series bins. They arrive with a space, a list of things they want to store (dog leashes, umbrellas, shoes, bags), and a set of constraints (can't drill into the back panel, one side has a coat hook bar already installed). StorageMaxxing should generate candidate approaches — bin systems, hanging systems, open modular grids — and let the user compare them.

### 2.4 Access Orientation Is a Fundamental Constraint

Every space has an **open face** — the direction from which a user reaches in. This is not a cosmetic detail; it structurally changes what configurations are physically useful:

**Top-down access** (drawers, trays, open-top containers): the user looks down and can reach anywhere in the footprint. Depth is additional capacity. A 24″×24″ drawer is fully accessible regardless of fill. The packer can use the full footprint.

**Front-access** (shelf bays, cubbies, cabinet interiors): the user reaches in through the open face. Anything behind the first row requires moving the front row first — which is unacceptable for daily-use storage. The effective _accessible_ depth is therefore one bin deep, not the full shelf depth.

This drives hard rules in the packing engine (§7.0) and shapes which options are reasonable to generate (§8). The open face must be captured as a first-class attribute of every SpaceTemplate — it cannot be inferred from dimensions alone.

_Example: a 14″-deep shelf bay with Akro-Mils 30020 bins (4⅛″ deep) could theoretically place three rows front-to-back. Only the front row is accessible without rearranging. The packer defaults to single-row fill; two-row configurations require explicit user override and are flagged "reduced accessibility."_

## 3. Users & Personas

### 3.1 Primary Personas

**The Workshop Owner**
Has 1–3 tool chests with 6–12 drawers each, plus open shelving for bulk storage. Shops Schaller for drawer organization, Akro-Mils for shelf bins. Primary pain: manually configuring each drawer differently, tracking what goes where, generating a purchase list before ordering. Values speed and accuracy. Knows what bins they want — needs help with quantities and layout.

**The Facilities / Lab Manager**
Manages storage across a shared space — makerspace, lab, light manufacturing floor. Needs to configure multiple storage units of different types, track total bin inventory, and generate purchase orders. Needs aggregate BOM across an entire zone. Will pay meaningfully for time saved versus spreadsheet work.

**The Home Organizer with Irregular Spaces** _(new in v0.3)_
Has real furniture with non-standard geometry — entryway organizers, pantry shelves, closet systems, garage cabinets — that don't map cleanly to standard bin formats. Doesn't know exactly what bins they want; needs to discover options. Has practical installation constraints: can't drill into a wall or back panel, shelves have weight limits, certain areas need to stay accessible.

_Concrete example: an entryway piece with a top shelf, two large center cubbies separated by a vertical divider, and a row of small shoe slots at the bottom. The user needs to store dog leashes, poop bags, umbrellas, spare shoes, and miscellaneous bags. The back panel is particle board (no drilling). A metal coat bar is already installed in one cubby but barely used. The space is currently ~70% wasted. What are the options?_

**The Maker / DIY Builder** _(new in v0.3)_
3D prints their own bins, cuts acrylic panels, or builds custom drawer inserts. Uses open modular systems like Gridfinity or OpenGrid as a base. Wants to model their custom components and see how they tile into a space, alongside or instead of commercial options. May have STEP files or parametric models they want to reference.

### 3.2 Underserved Scenarios

- **Mixed-use spaces:** A space that needs to serve multiple purposes (coats + dog supplies + shoes in an entryway) requires planning across object categories, not just bin sizes
- **Installation-constrained spaces:** Can't drill, max weight per shelf, specific wall anchors only — these are planning constraints as real as bin dimensions
- **Inherited / partially-filled spaces:** User has some existing bins and wants to optimize what remains
- **Comparison planning:** "Show me what this looks like with Akro-Mils vs Gridfinity vs a custom solution" — side-by-side
- **Tolerance-sensitive fits:** User isn't sure if a bin "squishes" slightly; needs to know minimum clearance, not nominal fit

### 3.3 Non-Goals (User Scope)

- Professional warehouse logistics (different scale, different software category)
- Shipping/moving container packing (weight, routing, fragility not in scope)
- Retail planogram design (existing specialized software)
- Furniture design (StorageMaxxing models spaces inside furniture, not the furniture itself)

## 4. Core Concepts & Data Model

### Type Hierarchy Clarification

Three catalog item types exist. Their relationship must be explicit:

- **`BinSpec`** — commercial or user-defined bin with inch dimensions. Used by Schaller, Akro-Mils, CSV imports.
- **`GridUnit`** — a bin or container for a modular grid system. Sized in grid units, not inches. Extends the concept of BinSpec but stores grid coordinates, not raw dimensions.
- **`GridAccessory`** — a non-bin grid item (hook, label, blank plate). Occupies grid cells but does not hold items inside.

`SpaceConstraint.binId` can reference any of these three types. The packer selects the correct placement algorithm based on the item's type and the space's system. BOM generation treats all three uniformly (quantity × price).

---

### 4.1 BinSpec

A product in a vendor catalog or user-defined library. Immutable — users reference BinSpecs, they don't edit them within a project.

```typescript
interface BinSpec {
  id: string;
  name: string;
  sku: string;
  vendor: string; // "schaller" | "akromils" | "gridfinity" | "user_defined" | etc.
  catalogSource: "builtin" | "csv_import" | "user_defined" | "open_system";
  system: StorageSystem; // which modular system this belongs to, if any
  w: number; // nominal width, inches
  l: number; // nominal length/depth, inches
  h: number; // height, inches
  toleranceW?: number; // dimensional tolerance ±, inches (default 0)
  toleranceL?: number;
  toleranceH?: number;
  price: number; // per-unit USD; 0 for open/free systems
  priceApproximate: boolean;
  colors: string[];
  stackable: boolean;
  stackMaxTiers: number | null;
  hangable: boolean;
  railCompatible: boolean; // fits standard rail/pegboard hook systems
  gridUnit?: number; // for modular systems: base grid unit in mm (Gridfinity = 42mm)
  compatibleSystems: SpaceType[];
  installationRequirements: InstallationRequirement[];
  modelUrl?: string; // URL to STEP/OBJ/GLB for 3D preview
}

interface InstallationRequirement {
  type: "drill" | "rail" | "adhesive" | "freestanding" | "stackOnly";
  description: string;
}
```

### 4.2 GridUnit and GridAccessory

Modular grid systems (Gridfinity, OpenGrid) require two distinct catalog item types that are not well-served by BinSpec alone.

**GridUnit** is a bin or container that occupies grid cells. Characterized by its size in grid units (not inches) and height.

**GridAccessory** is a non-bin item that occupies grid cells: hooks, label holders, blank plates, shelf dividers, cable clips. These are not bins — they don't hold other items inside them — but they compete for grid cells and appear in the BOM.

```typescript
interface GridUnit {
  id: string;
  name: string;
  system: "gridfinity" | "opengrid" | "custom_grid";
  gridW: number; // width in grid units (e.g. 1, 2, 3)
  gridL: number; // length/depth in grid units
  gridH: number; // height in system height units (Gridfinity: multiples of 7mm)
  // Actual dimensions computed from system constants
  // Gridfinity: actualW = gridW × 42mm - 0.5mm tolerance
  price: number; // 0 for open designs; user can set material cost
  stackable: boolean;
  catalogSource: "builtin" | "user_defined";
  modelUrl?: string;
}

interface GridAccessory {
  id: string;
  name: string;
  system: "gridfinity" | "opengrid" | "custom_grid";
  type:
    | "hook"
    | "label"
    | "divider"
    | "blank"
    | "shelf"
    | "cable_clip"
    | "custom";
  gridW: number;
  gridL: number;
  // Height: typically 1 grid height unit
  price: number;
  catalogSource: "builtin" | "user_defined";
}
```

**Packing implication:** When the user selects a grid system for a space, the packer allocates a baseplate covering the available grid cells, then places GridUnits and GridAccessories onto the baseplate. Remaining unoccupied grid cells show as "unused" in the BOM.

### 4.3 InstallationConstraint

_New in v0.3._ A constraint on what can be done to or inside a Space. Installation constraints filter which BinSpecs are compatible before packing begins.

```typescript
interface InstallationConstraint {
  type:
    | "noDrill"
    | "noAdhesive"
    | "maxWeightLbs"
    | "noWallMount"
    | "railPresent"
    | "custom";
  value?: number; // e.g. maxWeightLbs = 25
  surface?: "back" | "sides" | "top" | "floor";
  notes?: string; // "back panel is particle board"
}
```

**Effect on packing:**

- `noDrill` on `back`: eliminates all bins requiring wall/back mounting
- `maxWeightLbs`: reduces auto-fill quantity ceiling to stay under weight limit
- `railPresent`: unlocks hang-mounted bins as an option in that space

### 4.3 SpaceTemplate

A reusable space geometry definition. Supports non-rectangular cross-sections via a simple polygon.

```typescript
type AccessFace = "top" | "front" | "top+front" | "all-sides";

interface SpaceTemplate {
  id: string;
  name: string;
  type: SpaceType;
  accessFace: AccessFace; // REQUIRED — determines packing rules (see §7.0)
  // Simple rectangular spaces (most cases)
  w?: number;
  l?: number;
  h?: number;
  // Non-rectangular spaces (polygon footprint)
  footprint?: Point2D[]; // ordered polygon vertices in inches
  // Derived
  gridResolution: number; // default 0.5"
  packingModel: "2d" | "2.5d" | "3d";
  installationConstraints: InstallationConstraint[];
  // Geometry modifiers
  dividers?: Divider[]; // internal vertical or horizontal dividers
  obstacles?: Obstacle[]; // fixed objects (existing coat bar, pipe, etc.)
}

interface Divider {
  axis: "x" | "y" | "z";
  position: number; // inches from origin
  thickness: number;
}

interface Obstacle {
  x: number;
  y: number;
  z: number;
  w: number;
  l: number;
  h: number;
  label: string; // "coat bar", "plumbing pipe"
  permanent: boolean;
}
```

### 4.4 SpaceInstance

A specific occurrence of a SpaceTemplate in an Assembly. Has its own constraint set.

```typescript
interface SpaceInstance {
  id: string;
  templateId: string;
  name: string;
  count: number;
  constraints: Map<BinSpecId, SpaceConstraint>;
  activeStrategy?: PackingStrategyId; // which options-mode strategy is selected
  placedBins?: PlacedBin[];
}
```

### 4.5 SpaceConstraint

The planning primitive. Applied per bin type per space instance.

```typescript
interface SpaceConstraint {
  binId: string;
  lo: number; // minimum count (0 = no floor)
  hi: number | null; // maximum count (null = unlimited auto-fill)
  hard: boolean;
  color: string;
  mode: "off" | "auto" | "soft" | "hard";
}
```

**Constraint modes:**

| Mode   | lo  | hi   | hard  | Behavior                                   |
| ------ | --- | ---- | ----- | ------------------------------------------ |
| `off`  | 0   | 0    | —     | Excluded from this space                   |
| `auto` | 0   | null | false | Fill as many as fit                        |
| `soft` | N   | M    | false | Try to place N–M; warn if not met          |
| `hard` | N   | M    | true  | Must place N; layout invalid if impossible |

### 4.6 Unit

A physical furniture structure containing one or more SpaceInstances.

```typescript
interface Unit {
  id: string;
  name: string;
  type:
    | "toolChest"
    | "bookshelf"
    | "cabinetSystem"
    | "garageShelving"
    | "entryway"
    | "custom";
  outerW: number;
  outerH: number;
  outerD: number;
  wallThickness: number;
  pillarWidth: number;
  backWallThickness: number;
  backMaterial?: "wood" | "particleboard" | "metal" | "drywall" | "open";
  spaces: SpaceInstance[];
  shelfPositions?: number[];
  modelUrl?: string; // STEP/GLB import for 3D visualization
}
```

### 4.7 Assembly

A named collection of Units and/or standalone SpaceInstances sharing a planning context.

```typescript
interface Assembly {
  id: string;
  name: string;
  units: Unit[];
  standaloneSpaces: SpaceInstance[];
  aggregateConstraints: AggregateConstraint[];
  globalLimits: Map<BinSpecId, number | null>;
  storageCategories?: StorageCategory[]; // "shoes", "dog supplies", "umbrellas"
}

interface AggregateConstraint {
  binId: string;
  minTotal: number;
  maxTotal: number | null;
  hard: boolean;
}

interface StorageCategory {
  id: string;
  name: string; // "dog leashes", "umbrellas"
  requiredCount: number;
  notes: string;
}
```

_StorageCategories are purely informational. They are a checklist for the user, not inputs to the solver. The packer has no concept of "a leash needs a hook" — it only knows bin dimensions and counts. The checklist in Options Mode (§8.4) is user-maintained: the user marks which items are served by which bins after seeing the layout. Future AI-assisted matching (Phase 3+) could suggest bin-to-item mappings, but that is not in scope for Phase 0–2._

### 4.8 Project

Top-level grouping. Contains multiple assemblies.

```typescript
interface Project {
  id: string;
  name: string;
  assemblies: Assembly[];
  budgetCap: number | null;
  notes: string;
}
```

## 5. Storage Systems & Vendor Catalog

### 5.1 System Taxonomy

StorageMaxxing supports three classes of storage system:

**Commercial vendor systems** — products with SKUs, prices, and physical availability. Schaller, Akro-Mils, Lista, Husky, etc.

**Open modular systems** — free/open designs typically 3D-printed or cut from acrylic. No per-unit cost (or user-specified material cost). Key examples: Gridfinity, OpenGrid. These are fully parameterized by a base grid unit.

**User-defined systems** — custom dimensions and pricing entered by the user or imported via CSV or 3D model file. Covers everything from custom acrylic bins to bins sourced from niche vendors not in the catalog.

### 5.5 Gridfinity _(Phase 1)_

Open-source modular bin system by Zack Freedman. Designed for 3D printing. Base unit: 42mm × 42mm grid. Users stack and combine grid modules.

```typescript
const GRIDFINITY_BASE_MM = 42;
const GRIDFINITY_BASE_IN = 42 / 25.4; // 1.654"

// A 3×2 Gridfinity bin occupies:
// w = 3 × 42mm - 0.5mm tolerance = 125.5mm ≈ 4.94"
// l = 2 × 42mm - 0.5mm tolerance = 83.5mm ≈ 3.29"
// Heights: multiples of 7mm, typically 1–6 units tall
```

**Implementation notes:**

- BinSpecs for Gridfinity are parameterized by grid size (WxL in units) and height (in 7mm increments)
- Price is $0 (open design) or user-specified material cost per print
- Catalog entry should cover common sizes: 1×1, 1×2, 2×1, 2×2, 1×3, 3×1, 2×2, 2×3, 3×3
- Tolerances: ±0.5mm fit tolerance should be modeled to avoid layout showing a fit that doesn't physically work

### 5.6 NeoGrid & openGrid — Parametric Divider Systems

NeoGrid is a hybrid system that uses 3D-printed connectors (X, T, L, I) combined with sheet material (3mm plywood, MDF, or acrylic) to create custom-sized compartments.

**Core Grid Standards:**

- **openGrid (Primary):** 28mm unit. Higher precision for tight spaces.
- **Gridfinity:** 42mm unit. Industry standard for small parts bins.
- **Interoperability:** 3 openGrid units (84mm) = 2 Gridfinity units (84mm). The systems can share a space using common baseplate adapters.

**Packing Logic (Divider-based):**

Unlike fixed-size bins, NeoGrid "fills" a space by placing connectors at grid intersections and sliding cut-to-size dividers into slots. The packer treats these as virtual bins with dimensions that are multiples of 28mm or 42mm.

- **Standard Divider Thickness:** 3mm (0.118″).
- **Nominal Footprint:** Multiples of 28mm (e.g., 56×112mm) or 42mm.
- **Parametric Height:** Custom cut to match `spaceH` minus connector clearance (~2mm).

### 5.7 OpenGrid _(Phase 1)_

Open modular panel system for wall/pegboard organization. Panels tile on a grid; hooks, bins, and accessories slot in. Particularly useful for wall spaces and the back of cabinets — including particle board backs with adhesive mounting.

- Base panel: 75mm × 75mm grid
- Accessories: hooks, small bins, shelf brackets, label holders
- Can be installed without drilling (adhesive strips for lighter loads)
- Relevant for the entryway cubby use case — panels can line the back wall without penetrating it

### 5.10 Storage Systems: Bin-Filling vs. Drawer-Filling

The core engine supports two primary optimization product areas using the same primitive CAD interface:

**Drawer Filling (Footprint-first):**
- **Goal:** Maximize floor utilization.
- **Primary Systems:** Schaller, Gridfinity, NeoGrid.
- **Constraint:** `top` access orientation. Everything must be reachable from above.
- **Constraint Engine Role:** Optimize 2D placement and quantity targets across multiple drawers.

**Bin Filling (Volume-first):**
- **Goal:** Sub-divide a larger container or shelf bay into logical compartments.
- **Primary Systems:** NeoGrid dividers, Akro-Mils Divider Boxes, custom partitions.
- **Constraint:** Often involves `front` or `top+front` access.
- **Constraint Engine Role:** Validate that the sub-division is feasible given the items being stored (StorageCategories) and the physical grid limits.

## 6. Space Modeler

The modeler must feel like sketching, not engineering. A user with a tape measure should reach a fully planned layout in under 5 minutes.

### 6.1 Input Modes

#### Quick Entry _(Phase 0 — Primary)_

Dimension inputs support fractions (`15 3/4`, `16 1/8`) and commit on blur/Enter. Space type picker with icons. Immediately available for planning.

#### Unit Builder _(Phase 0)_

For structured furniture. User defines outer envelope and internal structure; usable bay dimensions computed.

- **Bookshelf:** outer W×H×D, wall/pillar widths, shelf positions → bay dims
- **Tool chest:** outer dims, drawer count, per-drawer height → drawer dims
- **Garage shelving:** outer envelope, shelf count, mid-span columns
- **Entryway unit:** outer dims, vertical dividers, shelf positions, back material → multiple bays

Back material should be captured here because it determines which installation constraints apply automatically (particle board back → `noDrill` on back surface set by default).

#### Sketch Modeler _(Phase 1)_

2D canvas for irregular or non-standard spaces:

- Rectangle and polygon drawing tools
- Dimension annotation on any edge (overrides drawn length)
- Snap-to-grid (0.5″, 1″, 2″, 6″)
- Obstacle placement (fixed objects: existing coat bars, pipes, etc.)
- Divider placement tool (internal partitions)
- Extrude footprint to 3D volume by setting height

The sketch modeler does not need to be parametric in Phase 1. Parametric relationships are Phase 2+.

**Sketch → Extrusion → Fill pipeline:**

The intended workflow is:
(1) draw a 2D footprint representing the **Void Geometry** on a grid canvas,
(2) annotate edges with measured dimensions,
(3) set height to extrude the footprint into a 3D volume,
(4) define the **Fill Volume(s)** within that void by selecting the Gravity and Access faces,
(5) optionally add **Keep-Out Zones**, and
(6) feed this result (a `SpaceTemplate`) into the system selection and packing workflow.
This is the spatial equivalent of: tape measure → sketch → configure.
Steps 1–3 live in the Sketch Modeler; Step 4 is the existing Configure/Options flow.
This allows algorithms to easily apply constraints to the designated fill spaces before dealing with complex solid modeling of the furniture itself.

#### STEP / 3D Model Import _(Phase 2)_

- Accept STEP, OBJ, GLB files
- Extract bounding box as default space geometry
- Allow user to refine interior dimensions
- Render actual furniture geometry in the 3D view alongside bins
- Critical for users who already have furniture models or manufacturer-provided files

## 7. Packing Engine

### 7.0 Feasibility & Architecture Decision

**Fully client-side is viable. No backend required.**

The packing problems in StorageMaxxing involve two distinct computational tasks that require different tools:

| Task                                                              | Right tool                    | Why                                                                      |
| ----------------------------------------------------------------- | ----------------------------- | ------------------------------------------------------------------------ |
| **Placement** — where exactly does each bin go?                   | MaxRects BAF heuristic        | Sub-millisecond, 85–95% of optimal, synchronous, well-tested             |
| **Constraint validation** — can this constraint set be satisfied? | GLPK.js (WASM) via Web Worker | Handles aggregate/global limit feasibility; async so it doesn't block UI |

**Why not ILP for placement?** A naive ILP formulation of the placement problem has O(bin*types × valid_positions × rotations) variables. For a 24″×24″ drawer at 0.5″ resolution that's ~140,000 binary variables — too large for GLPK.js to solve in real-time. The smart formulation (optimize over \_counts* only, then hand counts to MaxRects for placement) drops to ~80 variables, which is fast, but at that point the solver is barely doing anything MaxRects doesn't already handle. MaxRects is the right tool for placement.

**Why GLPK.js for constraint validation?** The genuinely hard constraint problems are:

- "Can these aggregate constraints be satisfied across 6 drawers with these bin types and global limits?" — a proper feasibility check
- "Is there a conflict between hard minimums and global caps?" — detectable as UNSAT
- These have small variable counts (one integer per bin type per space) and are fast for any MIP solver

**Solver selection:**

- **Phase 0:** GLPK.js — mature, MIT licensed, [battle-tested in browser](https://github.com/jvail/glpk.js), handles the constraint validation problems
- **Migration path:** HiGHS.js if GLPK proves slow for larger assemblies (more spaces, more bin types)
- **Watch list:** OR-Tools is developing a WASM/JS port — check status before Phase 1 implementation. If available and stable, it would replace GLPK.js entirely with CP-SAT, which is significantly more powerful for constraint satisfaction.

**Web Worker requirement:** WASM solvers are async and can take 50–500ms on larger problems. They must run in a Web Worker. The UI sends constraint state to the worker; the worker sends back feasibility results and suggested counts. `useMemo` is not appropriate for this layer — use `useEffect` + worker message passing with debounce.

```
Main thread                        Web Worker
────────────────                   ────────────────────────────────
constraint state  ──[postMessage]─▶  GLPK.js feasibility check
                                      → solve: can constraints be met?
placed bins       ◀─[postMessage]──  result: {feasible, suggestedCounts, conflicts}
  (MaxRects runs
   synchronously
   on main thread)
```

### 7.1 Access Orientation Rules _(applies to all packing models)_

Every space has an `accessFace` — the direction a user reaches in. This changes which placements are valid before any algorithm runs.

| Access Face | Effective depth for placement                                           | Notes                                               |
| ----------- | ----------------------------------------------------------------------- | --------------------------------------------------- |
| `top`       | Full space depth (L)                                                    | Drawer/tray — reach anywhere from above             |
| `front`     | One bin deep + 1″ clearance by default                                  | Shelf/cubby — second row requires explicit override |
| `top+front` | One bin deep for items taller than spaceH/2; full depth for short items | Open shelf                                          |
| `all-sides` | Full volume                                                             | Freestanding containers                             |

**One-row rule enforcement (front-access):**

```
effectiveDepth = min(spaceL, binL + 1.0)  // 1" clearance behind front row
// MaxRects grid is initialized with width = spaceW, depth = effectiveDepth
// User can override: uses full spaceL but affected bins flagged "reduced accessibility"
```

**Default `accessFace` by space type:**

| Space Type  | Default      | Rationale                                |
| ----------- | ------------ | ---------------------------------------- |
| `drawer`    | `top`        | Pulled out, accessed from above          |
| `shelfBay`  | `front`      | Open face only                           |
| `openShelf` | `top+front`  | Top and front both accessible            |
| `openFloor` | `all-sides`  | Walk around                              |
| `wall`      | `front`      | Wall-mounted is always front-access      |
| `custom`    | **Required** | Must be set explicitly — no safe default |

### 7.2 Placement Algorithm — MaxRects BAF

**Algorithm:** MaxRects Best Area Fit (Jylänki 2010) on a configurable grid (default 0.5″).

MaxRects is not a custom implementation — it is a standard published algorithm that achieves near-optimal packing density for 2D bin placement. It is fast, deterministic, and well-understood. We implement it once; we do not replace it with a solver.

**Four-phase placement order:**

1. Hard minimums — placed first; layout `invalid` if any fail
2. Soft minimums — best-effort; layout `partial` if unmet
3. Capped fills — fill to `hi` ceiling, largest-footprint-first
4. Unlimited auto-fill — fill remaining space, largest-footprint-first, loop until no more fit

**Tolerance:** Effective footprint for collision detection = `(bin.w + bin.toleranceW) × (bin.l + bin.toleranceL)`. Visual render uses nominal dimensions.

### 7.3 Grid System Placement — Gridfinity / OpenGrid

Grid systems snap to a fixed baseplate. The problem is much smaller than free placement and _can_ use the solver directly for placement because the grid cell count is small.

```
// Gridfinity: 42mm unit = 1.654"
// 24" drawer → floor(24 / 1.654) = 14 grid columns × 14 rows = 196 grid cells
// With ~12 bin types × 2 rotations × ~100 valid positions: ~2,400 variables
// → GLPK.js solves this in < 50ms
```

For grid systems, GLPK.js handles placement directly (not just constraint validation), because the grid-aligned problem is small enough. This is an exception to the two-layer split above.

**Grid remainder** (space at edges that doesn't fill a complete grid unit) is always computed and reported as a metric. It is not filled with other bin types in Phase 0.

### 7.4 Shelf Bay Packing — 2.5D

A shelf bay is modeled as a sequence of horizontal layers. Each layer runs §7.2 independently. Additional cross-layer constraints:

- Layer heights sum ≤ bay clearance H
- Stacking: if `bin.stackable` and same bin placed in layer k, it may stack in layer k+1 up to `stackMaxTiers`
- Front-access one-row rule (§7.1) applied per-layer: only the front row of each tier is accessible
- Tilt-access warning: when `binH > bayH × 0.85`, flag in layout view

**Mixed-height tiers:** When bins of different heights share a bay, compute all viable height combinations that tile the bay, pick the combination that minimizes unused height.

### 7.5 Constraint Validation — GLPK.js (Web Worker)

This is where the solver is used. The constraint validation problem is small (one count variable per bin type per space, plus aggregate and global limit constraints) and runs async in a Web Worker.

**Inputs to solver:**

```
for each space s:
  for each bin type i:
    count[i,s] ∈ ℤ≥0
    max_possible[i,s] = floor(effectiveArea(s) / area(bin[i]))  // upper bound from geometry

// Hard minimum constraints
∀ i,s where hard=true:  count[i,s] ≥ lo[i,s]

// Maximum constraints
∀ i,s where hi≠null:  count[i,s] ≤ hi[i,s]

// Global limits
∀ i with global cap G:  Σ_s count[i,s] ≤ G

// Aggregate constraints (assembly level)
∀ aggregate A:  Σ_s count[A.binId, s] ≥ A.minTotal

// Objective: maximize total placed area (as feasibility signal, not primary goal)
Maximize: Σ_i Σ_s count[i,s] × area(bin[i])
```

**Solver outputs → PlacementTargets → MaxRects**

The solver returns `count[i,s]` targets. MaxRects uses these as the quantity inputs for the four-phase placement algorithm. This is the handoff between the two layers.

**Explicit conflict states:**

| State                   | Meaning                                                        |
| ----------------------- | -------------------------------------------------------------- |
| `HARD_MIN_UNMET`        | Solver proves hard minimum geometrically impossible            |
| `AGGREGATE_UNMET`       | Assembly aggregate can't be met after all spaces packed        |
| `GLOBAL_LIMIT_CONFLICT` | Global cap < sum of hard minimums — detectable without solving |
| `INCOMPATIBLE_BIN`      | Bin dimension or installation requirement excludes it          |
| `INSTALLATION_BLOCKED`  | Bin requires drilling; space has `noDrill`                     |

### 7.6 Validity Model

| State     | Meaning                            | User-facing                       |
| --------- | ---------------------------------- | --------------------------------- |
| `valid`   | All hard constraints met           | Green, BOM accurate               |
| `partial` | Soft constraints unmet; hard met   | Amber, warnings listed            |
| `invalid` | One or more hard constraints unmet | Red, failures listed, BOM flagged |

Invalid layouts are still shown — user sees what did and didn't fit. Export gated on acknowledgment.

## 8. Options Mode — Showing Possibilities

The core UX loop: define a space, see what different systems look like in it, pick one, then configure. Options Mode is what makes StorageMaxxing a discovery tool rather than just a calculator.

### 8.1 The System-Per-Layout Rule

**Options Mode shows the same space filled with one system at a time.** Each strategy card represents one coherent system: all Schaller, or all Gridfinity, or all Akro-Mils. Cross-system mixing within a single space is not generated by Options Mode — it's a Configure Mode choice the user makes deliberately after picking a direction.

This is intentional. Mixing systems in one space (half Schaller, half Gridfinity) is technically possible but hard to evaluate at a glance and hard to purchase. The options comparison is most useful when each option is a clean, purchasable thing with a clear BOM and a single vendor relationship.

### 8.2 Options Mode: Multi-Strategy Generation

**Input to Options Mode (what the solver gets when no constraints are set yet):**

When a user has defined a Space but set no bin constraints, Options Mode uses a default strategy per system: all bins in the system set to `auto` (unlimited fill, no minimums). This is the unconstrained "how much can I fit?" baseline. The user can then select a strategy and move to Configure Mode to add minimums, exclusions, and maximums.

If the user _has_ set some constraints (e.g., "I know I want at least 4 of the 2×4 bins"), Options Mode respects those as hard constraints and generates strategies that satisfy them, filling the remainder with each compatible system.

**Strategy generation:**

1. For each system compatible with this space type + installation constraints: run unconstrained fill (all bins `auto`); compute metrics
2. For the top 2 systems by utilization %: generate one variant — largest available bin sizes only ("fewer large") — and report metrics for that too
3. Each strategy is one system. No cross-system mixing.

**Strategy cards present objective metrics. The user decides what matters.**

The tool does not rank or recommend. It computes and displays:

| Metric                | What it measures                                                                                                                  |
| --------------------- | --------------------------------------------------------------------------------------------------------------------------------- |
| Space utilization %   | Filled area (or volume for 3D) ÷ total available                                                                                  |
| Total bin count       | Number of individual bins placed                                                                                                  |
| Total bin area        | Sum of all bin footprint areas (in²)                                                                                              |
| Total bin volume      | Sum of all bin _exterior_ volumes (in³) — actual interior volume requires wall thickness data not in catalog; treat as proxy only |
| Estimated cost        | BOM total at catalog prices                                                                                                       |
| SKU count             | Number of distinct bin types — simpler = easier to order and manage                                                               |
| Grid remainder        | For modular systems: edge gap in inches that can't be tiled                                                                       |
| Constraints satisfied | Hard constraints met / total hard constraints                                                                                     |
| Accessibility         | Are all placed bins front-row accessible? (yes / no / partial)                                                                    |
| Installation required | What physical installation the system needs                                                                                       |

```
[ System: Schaller Corp — 2" depth ]
Utilization: 94%  ·  Bins: 34  ·  Bin area: 486 in²  ·  Vol: 851 in³
Cost: ~$28  ·  SKUs: 3  ·  All bins accessible  ·  No installation
[ View Layout ]  [ Select & Customize ]
─────────────────────────────────────
[ System: Gridfinity — 42mm grid ]
Utilization: 87%  ·  Bins: 24  ·  Bin area: 397 in²  ·  Vol: 694 in³
Print cost: ~$9  ·  SKUs: 4  ·  Grid remainder: 0.3"  ·  No installation
[ View Layout ]  [ Select & Customize ]
```

### 8.3 Entryway Cubby Example

For the space described in §3.1 (entryway with two large center cubbies, particle board back, existing coat bar in left cubby):

Options Mode would generate and score:

1. **Akro-Mils 30-series on shelf:** freestanding bins in both bays, no installation. Cost ~$30. Coverage ~65% (some depth wasted).
2. **OpenGrid panel system:** panel tiles adhesively mounted to back wall, accessories for leashes/bags/umbrellas. Cost ~$45 (panels + accessories). Coverage ~85% usable surface.
3. **Gridfinity baseplate:** 3D-printed grid in each bay, custom bins printed to fit items. Cost = print material ~$8. Coverage ~90%. Requires printer.
4. **Akro-Mils 30-series — Large bins only:** fewer but larger bins in both bays; better for bulkier items (umbrellas, bags). Cost ~$18. 65% utilization.

The user can see all four, click into any one, and move to Configure Mode to add minimums and exclusions. The existing coat bar is modeled as an obstacle in the space template — all strategies show it and work around it; the user decides whether to keep or remove it separately.

_Note: The "Akro-Mils + existing coat bar" framing from earlier drafts was cross-system mixing, which Options Mode doesn't generate. Obstacles (like the coat bar) are geometry constraints, not system choices._

### 8.4 "What Can I Store Here?" Checklist

When a user defines StorageCategories (§4.7), each strategy card shows a checklist:

```
✓ Dog leashes (hook on OpenGrid panel)
✓ Poop bags (small bin on panel)
✓ Umbrellas (tall hook or upright bin)
✗ Extra shoes — no floor space allocated in this strategy
```

This makes it visible when a strategy doesn't serve a stated need, before the user commits to it.

### 8.5 Options Mode vs. Configure Mode

| Mode               | Entry point                              | User intent | Output                        |
| ------------------ | ---------------------------------------- | ----------- | ----------------------------- |
| **Options Mode**   | "I have this space, what can I do?"      | Explore     | 3–5 strategy cards to compare |
| **Configure Mode** | "I know I want Akro-Mils, how many fit?" | Optimize    | Single live-updated layout    |

Users can toggle between modes. Selecting a strategy in Options Mode loads it into Configure Mode for refinement.

## 10. Technical Architecture

### 10.1 Phase 0 — Client-Side Only

No backend. All compute runs in <5ms for typical inputs. State serializes to localStorage. Zero infrastructure cost during validation.

### 10.2 Frontend Stack

- **React** — component model; `useState`/`useEffect` for constraint state; `useMemo` for synchronous MaxRects placement
- **MaxRects BAF** — placement algorithm, pure JS, runs synchronously on main thread, sub-millisecond
- **GLPK.js** — constraint validation solver, WASM, runs in Web Worker
  - [github.com/jvail/glpk.js](https://github.com/jvail/glpk.js) — MIT license, mature, battle-tested in browser
  - Used for: feasibility checks, aggregate constraint validation, global limit conflict detection
  - **Not** used for bin placement (too many variables; MaxRects handles placement)
  - Migration path: HiGHS.js if solve times become visible on larger assemblies
  - Watch: [OR-Tools WASM port](https://github.com/google/or-tools) — if production-ready by Phase 1, evaluate replacing GLPK.js
- **Web Worker** — GLPK.js runs in worker; main thread sends constraint state, worker returns feasibility + suggested counts
- **Three.js r128** — 3D visualization via CDN (`https://cdnjs.cloudflare.com/ajax/libs/three.js/r128/three.min.js`)
  - OrbitControls loaded separately (not bundled in r128)
  - `BoxGeometry` + `EdgesGeometry` for all rendering
  - No `CapsuleGeometry` (introduced r142)
- **No build step in prototype** — importmap or CDN imports

### 10.3 Modular System Grid Math

```typescript
// Gridfinity
const GRIDFINITY_UNIT_MM = 42;
const GRIDFINITY_UNIT_IN = GRIDFINITY_UNIT_MM / 25.4; // 1.6535"
const GRIDFINITY_HEIGHT_MM = 7; // per height unit
const GRIDFINITY_FIT_TOLERANCE_MM = 0.5; // per edge

// Grid bins: (gridW × gridL) units, (gridH) height units
// Actual footprint: (gridW × UNIT - tolerance) × (gridL × UNIT - tolerance)

// OpenGrid
const OPENGRID_UNIT_MM = 75;
const OPENGRID_UNIT_IN = OPENGRID_UNIT_MM / 25.4; // 2.953"
```

### 10.4 Tolerance Application

```typescript
function effectiveFootprint(bin: BinSpec): { w: number; l: number; h: number } {
  return {
    w: bin.w + (bin.toleranceW ?? 0),
    l: bin.l + (bin.toleranceL ?? 0),
    h: bin.h + (bin.toleranceH ?? 0),
  };
}
// Packer uses effectiveFootprint for collision detection
// Renderer uses nominal bin.w/l/h for visual display
```

## 11. Open Questions

### Resolved

**11.1 Depth usage for shallow shelf bins** — Front-fill in Phase 0; front/back row annotation in Phase 1.

**11.2 Sketch modeler placement** — Full-screen mode (separate route or modal). Sketching needs focus.

**11.3 Assembly aggregate constraint UI** — Separate "Assembly Rules" panel. Conceptually distinct from per-space constraints.

**11.4 Grid units** — Inches for Phase 0. Metric in Phase 1 (stored internally as inches).

**11.5 Incompatible bin display** — Show greyed with tooltip. Never hide silently.

**11.6 Pricing strategy** — Hard-coded with `priceApproximate: true` flag. User can override per-project. Open systems (Gridfinity) priced at $0 with optional material cost field.

### Unresolved

**11.7 How are identical shelf bays in the same unit synced?**
Options: always independent; "sync constraints" toggle that propagates across same-template bays. Recommendation: sync on by default, per-bay override. _Needs UX design before Phase 1._

**11.8 Grid resolution for large-format spaces**
0.5″ grid creates 36 cells per 18″ Akro-Mils bin width — correct but potentially slow for large floor spaces. May need adaptive resolution. _Needs performance profiling._

**11.9 Bins spanning multiple spaces**
A long bin spanning two shelf bays if a divider is removed. Not modeled. _Defer to Phase 2 or explicitly out of scope._

**11.10 How does Options Mode handle user-defined must-haves?**
If user says "I need at least 2 slots for umbrellas," how does Options Mode score strategies that can't satisfy this? Should unmet must-haves hard-exclude a strategy or just lower its score? _Needs design before Phase 1 Options Mode implementation._

**11.11 What is the right fidelity for STEP import geometry?**
Full mesh rendering in Three.js is expensive. Options: (a) render full mesh at reduced poly count; (b) extract bounding box only; (c) render bounding box but keep mesh for dimension extraction. _Needs Three.js performance testing before Phase 2._

**11.12 Which WASM solver — GLPK.js or HiGHS.js?**
GLPK.js is simpler to integrate and well-tested in browser environments. HiGHS.js is significantly faster on larger problems but is a newer WASM port with less community usage. For the problem sizes in Phase 0 (≤ 100 decision variables for a 24″×24″ drawer), either works. GLPK.js is the safe default; migrate to HiGHS.js if solve times become noticeable. _Decide at implementation start._

**11.13 How should Gridfinity remainder space be handled?**
When a space doesn't tile cleanly to the Gridfinity grid, the remainder may be usable for non-Gridfinity items. Should the packer try to fill grid remainders with other bin types, or always leave them empty? _Decision needed before Phase 1 Gridfinity implementation._

**11.13 How is access face communicated in the 3D view?**
The open face of a shelf or cubby is the most important spatial concept for understanding the layout — but Three.js renders walls as opaque planes that may hide it. Options: (a) always render the open face as absent/invisible; (b) render it as a distinct colored frame; (c) use a dashed line. _Needs visual design decision before Phase 1 3D view._

**11.14 In Options Mode, how many strategies is "enough" before the UI becomes overwhelming?**
With 5+ compatible systems and 2 variants each that's potentially 10 cards. Recommendation: cap at 5 cards maximum, ranked by score. But what's the right scoring weight between coverage, cost, and BOM simplicity? _Needs user research or initial assumption + iteration._

**11.15 How are GridAccessories (hooks, labels) shown in the layout view?**
Bins are boxes — easy to render. A hook is a thin rail-mounted piece with a specific 3D shape. In Phase 0/1, representing hooks as flat colored rectangles on the grid may be sufficient, but it could confuse users. _Render decision needed before Phase 1 grid accessories implementation._

:**11.16 What does "the same space with different systems" mean when system grid units don't align?**
A 15¾″ wide drawer shows 9 columns of Schaller 1.75″-wide bins, but only 9 Gridfinity cells (each 1.65″) with a 0.54″ remainder. These aren't the same number of usable positions. How should Options Mode visually communicate this difference? _Important for making the comparison fair and readable._

## 12. Out of Scope

Excluded from all phases to maintain product focus:

- Weight-based packing (fragility, load sequencing)
- Route-optimized logistics (truck loading, delivery sequencing)
- Retail planogram software (product facing, brand adjacency)
- Automated purchasing / e-commerce integration (BOM export is the handoff)
- Physical automation integration (bin picking, ASRS)
- Room-scale interior design (furniture placement)
- Thermal or environmental storage constraints (cold storage, humidity)
- Structural engineering (shelf load ratings are user-input constraints, not computed)
- Furniture design (StorageMaxxing models spaces _inside_ furniture, not the furniture itself)
